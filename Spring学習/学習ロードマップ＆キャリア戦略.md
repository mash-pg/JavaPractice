## 33歳・未経験という立場について

### 現実的なハンデ

- 20代と比べて「ポテンシャル採用」の猶予期間が短い
- 現場では年下の先輩・リーダーの下につく場面が多い
- 「未経験33歳」という目で最初は見られる

### ただし武器もある

| 武器 | 説明 |
|------|------|
| 社会人経験 | 報連相・納期意識・ビジネスマナーは新卒より圧倒的に上 |
| 覚悟 | 異業種から転職してきた人は学習意欲が高い傾向がある |
| Java Gold | 未経験で Gold を持っている時点で基礎力の証明になる |
| 年齢による落ち着き | パニックになりにくい、冷静に質問できる |

### 33歳未経験が意識すべきこと

**1. 最初の1年で「未経験」のレッテルを剥がす**

- 現場で「この人は使える」と思わせるのが最優先
- 技術力だけでなく、タスク管理・報告・質問の質で差がつく
- Java Gold + Spring Boot の自主学習実績は初日から武器になる

**2. 年下の先輩にも素直に学ぶ**

- 技術の世界では年齢よりスキルと経験が優先される
- 変なプライドを持たず「教えてください」と言えることが重要
- 前職の経験を活かせる場面では積極的に貢献する

**3. 年齢的に「ただのコーダー」で終わらない意識**

- 30代後半になると「手を動かすだけの人」は市場価値が下がる
- 早い段階から設計・レビュー・後輩指導の経験を積む
- 業務知識（ドメイン知識）を武器にするキャリアパスも視野に入れる

**4. 学習速度を落とさない**

- 入社後に学習を止める人が多い。33歳スタートでそれをやると追いつけなくなる
- 平日は現場の技術をキャッチアップ、休日は個人開発で新技術に触れるサイクルを維持する

## 省略した技術について

### Thymeleaf（テンプレートエンジン）

- サーバーサイドでHTMLを生成するための技術
- ロードマップはREST API（JSON返却）に集中しているので不要だった
- 今の開発現場ではフロントは React/Vue/Next.js 等で分離するのが主流
- **学ぶ優先度: 低め**。業務系Webアプリ（管理画面など）で使う場面はあるが、API開発を軸にするなら後回しでOK

### MyBatis（SQLマッパー）

- Spring Data JPA の代替技術。SQLを自分で書いてマッピングする
- JPA は「オブジェクト中心」、MyBatis は「SQL中心」
- 日本の現場ではMyBatisが多い（特にSIer系・業務系）
- **学ぶ優先度: 中〜高**。就職先・案件によっては JPA より使う機会が多い。ただしJPAを理解した上で学ぶと吸収が早い

### AOP（アスペクト指向プログラミング）

- ログ出力・トランザクション管理・認証チェックなど横断的関心事を分離する仕組み
- `@Transactional` や `@Slf4j` を使った時点で、裏ではAOPが動いている
- 自分でAOPを書く場面は限定的（共通ログ、実行時間計測など）
- **学ぶ優先度: 中**。仕組みを理解しておくと `@Transactional` が効かないバグの原因がわかるようになる

## シニアエンジニアとの会話力

### シニアがよく使うフレーズと意味

| フレーズ | 意味 |
|----------|------|
| 「それ、責務が違うよね」 | そのクラスがやるべき仕事じゃない → レイヤー設計の話 |
| 「N+1になってない？」 | DBを何回も叩く非効率なコード → JPA使用時に頻出 |
| 「破壊的変更になるけど大丈夫？」 | 既存の動作が変わる修正 → 影響範囲を確認しろ |
| 「冪等性は担保されてる？」 | 同じ操作を2回やっても結果が同じか → API設計の話 |
| 「その修正のスコープは？」 | どこまで直すの？ → 影響範囲の確認 |
| 「エビデンス取った？」 | テスト結果のスクリーンショットや記録 |

### 会話で信頼を得るコツ

```
× 「分かりません」（で終わる）
○ 「○○までは理解していますが、△△の部分が分かりません」
```

```
× 「できました」（だけ報告）
○ 「○○のパターンで動作確認しました。△△のケースはまだ未確認です」
```

自分が何を分かっていて何を分かっていないかを伝えられることがシニアから信頼されるポイント。

## SES 現場の技術環境について

### Java のバージョン問題

SES 現場では **古い Java バージョン** のプロジェクトがまだ多い。

| バージョン | 現場での状況 |
|-----------|-------------|
| Java 8 | 非常に多い。長期サポート（LTS）のため移行が進んでいない |
| Java 11 | 増えてきているがまだ少数派 |
| Java 17+ | 新規プロジェクトのみ。既存システムでは稀 |

### 実務で意識すべきこと

**Stream API（Java 8+）と拡張 for 文の両方を使えるようにする**

```java
// Stream API（モダン）
int total = transactions.stream()
    .mapToInt(t -> t.getAmount().getValue())
    .sum();

// 拡張 for 文（クラシック）
int total = 0;
for (Transaction t : transactions) {
    total += t.getAmount().getValue();
}
```

| 観点 | Stream API | 拡張 for 文 |
|------|-----------|------------|
| 可読性 | 慣れれば高い | 誰でも読める |
| デバッグ | やや難しい | 簡単 |
| 現場での採用 | モダンな現場 | レガシー現場 |
| Java バージョン | 8 以上必須 | どのバージョンでも可 |

**ポイント**:
- 学習時は両方の書き方で実装してみる
- 現場のコーディング規約に合わせる柔軟性を持つ
- 「Stream しか書けない」は危険。どちらも書けるのがプロ

### その他の現場差異

| 技術 | モダン現場 | レガシー現場 |
|------|-----------|-------------|
| ビルドツール | Gradle | Maven（または Ant） |
| ORM | Spring Data JPA | MyBatis / 生 JDBC |
| IDE | IntelliJ | Eclipse |
| バージョン管理 | Git | SVN（まだある） |

現場に入ったら「郷に入っては郷に従え」。自分の好みより現場のルールを優先する。

## キャリアの成長段階

### レベル1：言っていることが分かる（現場 3〜6ヶ月）

- 用語が分かる、指示を正確に理解できる、質問が的確にできる
- 入社前の1ヶ月の準備で、ここへの到達はかなり早まる

### レベル2：技術的な議論に参加できる（1〜2年）

- 設計レビューで意見を言える、PRレビューで指摘できる
- 「こうした方がいいのでは？」と代替案を出せる

### レベル3：シニアと対等に議論できる（3〜5年）

- 設計判断を主導できる、技術選定の理由を説明できる
- トレードオフを理解した上で判断を下せる

### 伸びる人の特徴

- 現場のコードを読みまくる（自分の担当外も）
- 「なぜこの設計なのか」を先輩に聞く
- 障害対応に自分から首を突っ込む
- 個人開発を続ける

### 伸びない人の特徴

- 言われたタスクだけやる
- 同じ技術スタックの案件に居続ける
- 「仕事で使わないから」と学習を止める
